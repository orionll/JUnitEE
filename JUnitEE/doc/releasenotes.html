Release Notes for JUnitEE 1.5.1
 
Changes since release 1.5 final

The changes since release 1.5 final are:

  • fix for bug 623737 (Ant task not working when using runall in combination with the resource attribute
  • fix for bug 624812 (error/failure property is not set by the Ant task)

 
Changes since release 1.5rc2

The changes since JUnitEE 1.5rc2 are:

  • xml output is now 100% compliant to the xml test report generated by the JUnit Ant task
  • bug fix: NPE in JUnitEEServlet.searchForTests

 
Changes since release 1.5rc1

The changes since JUnitEE 1.5rc1 are:

  • request parameter/init parameter to specify stylesheet for xml output
  • JUnitEEXMLServlet which produces xml output by default
  • the system property 'cactus.contextURL' is set by JUnitEEServlet to support Jakarta Cactus test cases
  • bug fix: an error message is displayed if a test classes can not be loaded by the test runner

 
Changes since release 1.4

The changes since JUnitEE 1.4 are:

  • integrated user interface to start tests
  • improved test report with possibility to start test suites or single tests (e.g. for debugging)
  • test report can also be generated in XML format
  • new JUnitEEWar Ant task to build the test war

 
Migration

The communication protocol between the JUnitEE Ant task and the JUnitEE servlet has changed. Therefore the Ant task of release 1.4 will not work in combination with the servlet of release 1.5; also the Ant task of release 1.5 will not work with the servlet of release 1.4. Please always use the Ant task and servlet of the same release.

Because of refactorings and internal reorganizations junit.html.TestServletBase is marked as deprecated and should not be used anymore. Please replace any usage of this class by org.junite.servlet.JUnitEEServlet. junit.html.TestServletBase is still there and will work as expected but might be removed in the next release.

org.junite.servlet.JUnitEEServlet has a concrete implementation of the method getDynamicClassLoader which was abstract in former times. This default implementation answers the classloader by which the servlet class itself was loaded and therefore should work in any case you have your test cases packaged into a war file with JUnitEE. There are only some rare cases left that require you to create a subclass of org.junite.servlet.JUnitEEServlet and overwrite that method to return another classloader. So you might consider replacing your existing test servlet by using org.junite.servlet.JUnitEEServlet.
